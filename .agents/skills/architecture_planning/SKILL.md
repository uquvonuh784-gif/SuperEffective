# Навык планирования архитектуры

При проектировании новых функций или целых систем в этом проекте, ВСЕГДА следуй этим правилам:

1. **Анализ требований (PRD)**: Перед написанием любой строчки кода, внимательно изучи доступные требования (например, `task_tracker_prd.md`). Убедись, что цель и бизнес-логика полностью понятны. Если есть неясности — обязательно задай уточняющие вопросы пользователю.
2. **Проектирование "сверху вниз" (Top-Down)**: Начинай с общего видения системы. Определи основные компоненты (модули, сервисы, компоненты UI), их зоны ответственности и то, как они будут взаимодействовать друг с другом.
3. **Декомпозиция задач**: Разбивай большие задачи на маленькие, независимые и легко тестируемые шаги. Каждая подзадача должна иметь четкий результат (например, "Создать схему БД для задач", "Реализовать API получения списка", "Сверстать UI-компонент карточки").
4. **Обсуждение и утверждение**: Прежде чем приступить к масштабной реализации, предложи пользователю архитектурный план или план выполнения задач (в виде маркированного списка или простой схемы). Начинай писать код только после согласования.
5. **Чистая архитектура и принципы**:
   - Соблюдай принципы SOLID и DRY (Don't Repeat Yourself).
   - Разделяй бизнес-логику и отображение (UI).
   - Выделяй переиспользуемые компоненты и утилиты.
6. **Документирование решений (ADRs)**: Если принимается важное архитектурное решение (например, выбор библиотеки управления состоянием, структуры базы данных), кратко зафиксируй причину выбора, чтобы в будущем был понятен контекст.
7. **Язык общения**: Все предложения, планы и документация должны быть написаны на русском языке для удобства пользователя.
