# Навык написания кода

При написании или редактировании кода в этом проекте, ВСЕГДА следуй этим правилам:

1. **Читаемость превыше всего**:
   - Пиши код так, чтобы его мог понять другой разработчик (или ты сам через месяц) без дополнительных пояснений.
   - Избегай неочевидных сокращений в названиях переменных и функций (например, используй `getUserData` вместо `gud`, `userCount` вместо `uc`).
   - Названия функций должны быть глаголами (что делает функция?), а названия переменных и классов — существительными (что это такое?).

2. **Принципы проектирования**:
   - **KISS (Keep It Simple, Stupid)**: Не усложняй решения. Самый простой, работающий код обычно является лучшим.
   - **DRY (Don't Repeat Yourself)**: Избегай дублирования кода. Если логика повторяется более двух раз, вынеси её в отдельную функцию или компонент.
   - **YAGNI (You Aren't Gonna Need It)**: Не пиши функции "на будущее", которые не используются прямо сейчас. Пиши только то, что требуется для текущей задачи.

3. **Размер функций и компонентов**:
   - Функция должна выполнять только одну задачу (Принцип единственной ответственности).
   - Если функция становится слишком большой (например, больше 50-70 строк), подумай о том, чтобы разбить её на несколько более мелких и независимых функций.

4. **Комментарии и документация**:
   - Комментируй *почему* ты сделал именно так (особенно если это костыль или неочевидное решение), а не *что* делает код (код должен сам описывать *что* он делает).
   - Используй JSDoc/TSDoc для документации важных интерфейсов, типов и публичных функций, чтобы улучшить автодополнение в IDE.

5. **Обработка ошибок**:
   - Никогда не оставляй пустые блоки `catch`. Обязательно логируй ошибку или показывай пользователю понятное сообщение ("Что-то пошло не так").
   - Валидируй входные данные в функциях перед их использованием.

6. **Язык общения**:
   - Все обсуждения, предложения по коду и ревью должны быть на русском языке. Сами названия переменных/функций продолжаем писать на английском.
